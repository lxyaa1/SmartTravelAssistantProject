**建议采用 \*\*“契约优先 + 纵向切片 + 分模块扩展”\*\*，不要一上来让 agent 一次性生成完整项目。这个项目最容易失控的地方是：Agent、MCP、LangGraph 状态、结构化输出混在一起。**



**\*\*推荐工作流\*\***



**1. 先定义数据结构**  

&#x20;  **让 agent 先写 `TripRequest`、`TripPlan`、`PlanDay`、`VisitSlot`、`ValidationIssue` 这些 Pydantic schema。**  

&#x20;  **这是项目的“合同”，后面所有 Agent 和 MCP 都围绕这些结构工作。**



**2. 先做假 MCP**  

&#x20;  **不要一开始接真实天气、地图 API。先做 mock MCP server，比如固定返回：**

&#x20;  **- 某天大雨**

&#x20;  **- 某景点周一闭馆**

&#x20;  **- 某两个景点距离过远**  

&#x20;  **这样你能验证“发现问题 → 重规划”的闭环是否真的工作。**



**3. 做 LangGraph 最小闭环**  

&#x20;  **先只实现：**



&#x20;  **```text**

&#x20;  **parse\_request**

&#x20;    **→ plan\_route**

&#x20;    **→ collect\_data**

&#x20;    **→ validate\_plan**

&#x20;    **→ replan 或 final**

&#x20;  **```**



&#x20;  **不要先做复杂 UI、数据库、PDF 导出。**



**4. 再替换真实 MCP**  

&#x20;  **等闭环跑通后，再逐步把 mock MCP 换成真实 MCP：**

&#x20;  **- weather MCP**

&#x20;  **- map MCP**

&#x20;  **- attraction MCP**

&#x20;  **- transport MCP**



**5. 最后做 Streamlit UI**  

&#x20;  **UI 应该最后做，因为前期变化最大的其实是状态结构和工作流。**



**---**



**\*\*你可以这样指挥编码 Agent\*\***



**第一步，不要说“帮我完成整个项目”。可以这样下指令：**



**```text**

**请先不要实现完整业务逻辑。**

**请为一个本地旅游助手项目搭建 Python 项目骨架，使用 LangGraph + LangChain + MCP。**

**先完成：**

**1. 项目目录结构**

**2. Pydantic 数据结构**

**3. LangGraph 的空节点函数**

**4. 一个可以运行的最小 workflow**

**暂时不要接真实 API，MCP 工具可以先 mock。**

**```**



**第二步，专门让它做数据结构：**



**```text**

**请根据以下旅行规划需求设计 Pydantic schema：**

**用户输入包含出发地、目的地、日期、人数、预算、偏好、必去景点、避开项。**

**旅行计划需要包含每日行程、景点顺序、开始结束时间、交通方式、预计耗时、费用、备注。**

**验证问题需要包含问题类型、严重程度、发生日期、涉及地点、原因、建议修改方式。**

**要求所有 Agent 都使用这些 schema 作为输入输出，不允许自由文本在内部传递。**

**```**



**第三步，做 LangGraph 闭环：**



**```text**

**请实现 LangGraph workflow：**

**parse\_request → initial\_plan → collect\_mcp\_data → validate\_plan。**

**如果 validate\_plan 输出 high 或 critical 问题，则进入 replan；**

**replan 后再次进入 collect\_mcp\_data 和 validate\_plan。**

**最多循环 3 次。**

**如果没有严重问题或达到最大循环次数，则进入 final\_writer。**

**请把 iteration、current\_plan、plan\_versions、mcp\_results、issues 都保存在 state 中。**

**```**



**第四步，做 mock MCP：**



**```text**

**请实现本地 mock MCP servers，使用 stdio 方式运行。**

**先提供这些工具：**

**1. get\_weather(city, date)**

**2. get\_route\_time(origin, destination, mode)**

**3. get\_attraction\_detail(name, date)**

**4. search\_attractions(city, preferences)**



**mock 数据需要故意包含一些冲突：**

**例如雨天不适合户外景点、景点闭馆、路线耗时过长。**

**目标是验证 Agent 能发现问题并重新规划。**

**```**



**第五步，做验证 Agent：**



**```text**

**请实现 ValidatorAgent。**

**它接收 current\_plan 和 mcp\_results，输出结构化 ValidationIssue 列表。**

**必须检查：**

**1. 必去景点是否覆盖**

**2. 景点是否开放**

**3. 户外景点是否遇到恶劣天气**

**4. 每天总游玩时间是否过长**

**5. 景点间交通是否过远**

**6. 是否违反用户偏好或避开项**



**不要输出自然语言总结，只输出符合 schema 的结构化结果。**

**```**



**第六步，做重规划 Agent：**



**```text**

**请实现 ReplannerAgent。**

**它接收 current\_plan、ValidationIssue 列表、用户原始需求和 MCP 查询结果。**

**它只能针对问题修改计划，不要完全重写整个行程。**

**修改时需要保留用户的 must\_visit 景点，除非问题无法解决。**

**输出新的 TripPlan，并在 plan\_versions 中保存旧版本。**

**```**



**---**



**\*\*运行时智能体关键提示词\*\***



**`RoutePlannerAgent`：**



**```text**

**你是旅行路线规划智能体。**

**你的任务是基于用户需求生成一个初始旅行计划。**



**要求：**

**1. 输出必须符合 TripPlan schema。**

**2. 必须覆盖用户指定的 must\_visit 景点。**

**3. 每天行程不要超过 3-5 个主要景点。**

**4. 优先把地理位置接近的景点安排在同一天。**

**5. 不要假设天气、开放时间、交通耗时一定可行，这些之后会由 MCP 工具验证。**

**6. 不要输出解释性文本，只输出结构化计划。**

**```**



**`DataCollectorAgent`：**



**```text**

**你是 MCP 查询调度智能体。**

**你的任务不是制定计划，而是判断当前计划需要查询哪些外部信息。**



**你需要根据 current\_plan 生成 MCP 查询任务：**

**1. 每个旅行日期和城市需要查天气。**

**2. 每个景点需要查开放时间、门票、推荐停留时间。**

**3. 每两个连续地点之间需要查交通耗时。**

**4. 涉及跨城市移动时需要查交通方式。**



**输出必须是工具调用计划，不要评价行程。**

**```**



**`ValidatorAgent`：**



**```text**

**你是旅行计划可行性评估智能体。**

**你需要根据 current\_plan 和 MCP 查询结果找出具体问题。**



**只输出 ValidationIssue 列表。**

**严重程度分为 low、medium、high、critical。**



**high 或 critical 的例子：**

**- 必去景点没有安排**

**- 景点当天关闭**

**- 户外景点遇到暴雨**

**- 单日交通时间明显过长**

**- 行程时间冲突**



**不要直接重写计划。**

**```**



**`ReplannerAgent`：**



**```text**

**你是旅行计划修正智能体。**

**你需要根据 ValidationIssue 修改 current\_plan。**



**规则：**

**1. 优先局部调整，不要整体重写。**

**2. 优先调换日期，其次调整景点顺序，再其次替换非必去景点。**

**3. must\_visit 景点必须保留，除非 ValidationIssue 显示完全不可行。**

**4. 修正后仍然要符合用户偏好、预算和时间限制。**

**5. 输出新的 TripPlan schema。**

**```**



**`FinalWriterAgent`：**



**```text**

**你是旅行计划整理智能体。**

**你需要把最终 TripPlan 转换成用户可读的中文旅行方案。**



**输出包括：**

**1. 总体路线概览**

**2. 每日详细行程**

**3. 交通建议**

**4. 天气和开放时间提醒**

**5. 预算估算**

**6. 为什么这个方案可行**

**7. 仍需用户自行确认的事项**



**不要暴露内部 Agent 调度过程。**

**```**



**---**



**\*\*最重要的工程原则\*\***



**这个项目不要按“一个 Agent 一个文件然后各写各的”来推进，而是按闭环能力推进：**



**```text**

**能不能生成计划？**

**能不能查工具？**

**能不能发现问题？**

**能不能重规划？**

**能不能停止循环？**

**能不能输出最终方案？**

**```**



**先让这条链路完整跑通，再逐步增强每个模块。这样你很快就能得到一个可演示的版本，而且后面接真实 MCP 服务时不会推倒重来。**
